# 功能概述
- linux 平台下 c++项目中用于排查定位内存泄漏的模块。有简洁的接口、可复用、可调试、兼顾性能。


# 设计思路
- 对项目中动态内存申请操作附加审计功能。
    - 采用 LD_PRELOAD 替换 malloc/realloc/free 等函数的方法。
        - hook 内禁止再触发堆内存分配；
        - 暂不对 calloc 进行审计；
- 审计工作分为两部分：数据收集和结算。
    - 数据收集指的是，收集malloc/realloc/free等函数的调用者信息；
    - 结算指的是对收集到的数据进行处理（*alloc是借，free是还，还回的就销账，未还回的就还在账上）。
- 性能考虑
    - 因为审计必然要存储和处理大量信息，也就必然涉及访问一大块内存，一定会影响调用者的性能。审计工作之所以分成两部分，其实也是出于对性能的考虑。更进一步，是处于对数据收集工作的性能的考虑。
- 详述审计工作的两部分的想法
    - 数据收集
        - 数据封装为task，放入待结算队列，立即返回；
            - 队列不使用std中的queue(因为它的操作本身就包含了堆内存的申请和释放)，而是本模块中的独立组件。它是一个长度固定的循环队列。
        - task 简单分为两种类型，alloc和dealloc
            - malloc的审计类型是alloc，free的审计类型是dealloc
            - realloc 的审计简单处理，包含两个任务，一个dealloc和一个alloc（不考虑新旧指针是否相等）
        - 数据收集工作应足够快，如果不可避免要用到锁，那么一定是一个小范围的锁。
            - 多生产者和单一消费者的环境。生产者指的是需要审计的若干线程，消费者指的是本模块的结算线程。
        - 收集到的alloc应包含ptr,size,backtrace,单调递增的序号，时间戳(秒级)，调用者线程id
            - backtrace通过backtrace()获取栈指针即可(gdb可以翻译这些指针，配合c++filt即可得到可理解的栈帧。本模块不负责这些转换工作。)
            - 序号的用途。因为同一个ptr可能反复被申请/释放，而结算并不是同步操作的，序号用于保证结算时的顺序。
                - 在预期运行时长内假定序号不会发生回绕。
            - 时间戳的用途仅用于report（也许有助于观察alloc行为规律）
            - alloc和free是否有必要分两个任务队列？
                - 简化设计，共用一个task队列。
        - 收集到的dealloc task中仅需包含ptr
            - dealloc仅用于结算销账，不必包含更多复杂信息
            - 至于序号，此处没必要存储。因为对于相同ptr的反复申请/释放，仅需得出最后申请者即可。
        - 收集数据还应考虑队满的情况，应该覆盖最旧的，还是丢弃当前的？
            - 丢弃当前的，并计数被丢弃的counter。
    - 数据结算
        - 一个独立的常驻线程。审计开始时启动，直至被测进程结束；
        - 从task队列中取出任务并处理
            - 取task也许不可避免要加锁，应考虑对数据收集的性能的影响
            - 一次可以取出多个任务
        - 结算后的未销账的数据应存起来，等待后续销账
            - 未销账数据的信息与alloc任务中保留的信息一致
            - 数据满时丢弃alloc任务并计数
- 附加功能
    - 只考虑前述核心功能，cli接口（如实时开关、折叠backtrace、放行某些线程、只审计某些线程等）及其它功能暂不考虑
- 总结
    - 关于任务队列满和账本满时的处理，都有做考虑。这仅是出于代码逻辑严谨的做法，实际绝不能发生。
    - 在实际应用此模块功能时，若这两种情况任一发生，必须考虑改良设计(比如加大队列容量)或减轻被测目标负载。
        - 此时，在report中也必须说明数据不可靠。


# 组件设计
- 为便于代码维护，也为了分层避免循环依赖，应将模块功能拆分为若干组件。如：
    - MemoryAuditCommon.h，最底层的组件，包含公共数据类型；
    - 任务队列组件，一个class，专司任务队列管理，依赖MemoryAuditCommon.h；
        - 专为多生产者单消费者场景设计，内部不进行堆内存分配；
    - 小工具组件，其中包含malloc替换等hook(仅负责任务入队，不负责结算)，以及其它工具函数，依赖MemoryAuditCommon.h和任务队列组件；
    - 审计结算组件，包含一个独立线程，专司审计。依赖前述3个组件；
    - 模块API组件，依赖上述4个组件；
    - 各逻辑组件应分配合适的物理组件

# 接口设计

- 应包含以下接口
    - MallocAuditStart()
    - MallocAuditStop()
    - ReallocAuditStart()
    - ReallocAuditStop()
    - FreeAuditStart()
    - FreeAuditStop()
    - MemoryAuditReport()
- 接口说明
    - 内存释放操作必然晚于内存申请操作，有时时间差会稍长，因此设计分开 start/stop
        - 多次start等价于一次，stop同理。
    - 针对malloc和realloc分别提供接口是基于以下考虑
        - 单独审计其中之一，对数据收集性能减负
        - 单独审计其中之一，减少report结果，便于拆解排查内存泄漏
    - MemoryAuditReport
        - 仅在所有audit都是stop的状态下才工作
        - 内容输出到 ./leak_trace.log 中
        - 先输出「丢弃计数」，再输出账本
            - 若丢弃计数不为0，则说明本模块已超负荷，本报告的数据不完整，不可信。请考虑增加本模块处理能力，或降低被测系统的负载，直至丢弃计数为0(绝对不给用户不可靠的数据，否则仅供参考也会生出其它事端)。

                - 接下来逐行输出栈指针
